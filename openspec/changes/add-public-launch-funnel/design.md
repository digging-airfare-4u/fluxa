## Context

Fluxa 当前已经具备登录后完整的产品主链路：公开落地页、注册登录、AI 对话创作、画布编辑、作品发布、灵感发现、创作者主页、支付和邀请码激励。问题不在于“能力不存在”，而在于这些能力还没有被组织成一个面向正式发布的公开增长闭环。

现状约束：
- 发现页、作品详情页、创作者主页都位于 `/app/*`，而 [`src/app/app/layout.tsx`](/Users/cj/Documents/personal/project/fluxa/src/app/app/layout.tsx) 当前会强制未登录用户跳转 `/auth`。
- 公开首页 [`src/app/page.tsx`](/Users/cj/Documents/personal/project/fluxa/src/app/page.tsx) 视觉完成度较高，但更像品牌封面，缺少价值表达、示例内容入口和清晰转化承接。
- 分享入口已存在，但 [`src/components/share/ShareDialog.tsx`](/Users/cj/Documents/personal/project/fluxa/src/components/share/ShareDialog.tsx) 中“复制链接”仍指向占位链接，“分享对话图片”仍是 disabled。
- 社区已支持点赞、收藏、评论、关注，但缺少公开 UGC 在正式发布前所需的举报、限频和隐藏处理闭环。
- 当前观测能力偏工程日志，`src/lib/observability/metrics.ts` 仍是 console sink，不足以支持发布后漏斗分析和增长判断。

这个 change 是跨公开路由、SEO、创作入口、治理和分析的横切变更，适合先用 design 收敛关键边界，再进入 specs 和 tasks。

## Goals / Non-Goals

**Goals:**
- 让发现页、作品详情页、创作者主页在不破坏现有登录体验的前提下对未登录访客公开可访问。
- 为公开内容页提供可索引、可分享、可传播的 URL 与 SEO / Open Graph 元数据。
- 让用户从公开作品页一键进入 Remix/Fork，基于已发布快照快速创建自己的项目。
- 为公开社区补齐最小可用治理闭环，包括举报入口、限频、防滥用和运营隐藏处理。
- 为发布前后关键转化节点建立统一事件模型和生产可用的分析 sink。
- 将公开首页升级为转化首页，承接公开内容浏览与注册/开始创作。

**Non-Goals:**
- 不在本次改动中重写整个路由体系，不把所有 `/app/*` 页面都改成公开。
- 不做复杂推荐算法、精选运营后台或完整审核中台。
- 不做深度内容安全自动审核；首版以举报、限频和人工处理为主。
- 不做“分享图片卡片生成器”完整版；保留后续独立迭代空间。
- 不在本次改动中重构发布系统的底层快照模型。

## Decisions

### D1: 公开内容页采用“新增公开路由”，而不是直接解除 `/app` 登录保护

**选择**: 保留 `/app/*` 作为登录后产品域，同时新增公开路由承接发现页、作品详情页和创作者主页，例如 `/discover`、`/discover/[id]`、`/u/[userId]`。登录后的 `/app/*` 页面继续保留站内入口和互动体验，但公开分享、SEO 和外部访问统一落到公开路由。

**备选方案**:
- A) 直接放开 [`src/app/app/layout.tsx`](/Users/cj/Documents/personal/project/fluxa/src/app/app/layout.tsx) 对 discover/user 路由的鉴权
- B) 把所有社区页面完全迁出 `/app` 并删除站内版本

**理由**:
- A 会把当前“登录后产品壳”和“公开访问壳”耦合在一起，后续导航、顶栏、埋点和权限判断都会变复杂。
- B 改动面太大，容易引入站内回归。
- 新增公开路由能把“公开获客”和“登录后使用”明确拆层，风险最小，也更利于 SEO。

### D2: 公开路由的数据读取继续复用现有 publications 查询层，但明确“公开视图”与“登录增强态”分离

**选择**: 公开路由优先读取公开作品与公开资料数据；当用户已登录时，再增量读取点赞、收藏、关注等用户态信息。公开页面必须在无 session 的情况下完整渲染主要内容。

**备选方案**:
- A) 所有公开页统一走客户端加载，登录态和未登录态混在一起
- B) 为公开场景重新写一套独立查询模型

**理由**:
- A 不利于 SEO，也会让首屏和分享抓取变差。
- B 会重复已有发布系统查询能力。
- 复用现有 publication / profile query，同时分离“公开主体数据”和“登录增强态”，是最稳妥的演进方式。

### D3: SEO 和分享语义由公开详情页负责，分享入口统一复制公开 URL

**选择**: 公开首页、发现页、作品详情页、创作者主页补齐 `metadata` / `generateMetadata`，作品详情页输出标题、描述、封面、作者、canonical、Open Graph 和 Twitter Card 信息；编辑器 `ShareDialog` 的“复制链接”统一复制公开作品页链接，而不是 `/app/discover` 占位链接。

**备选方案**:
- A) 只做公开访问，不做 metadata
- B) 继续让分享入口复制站内链接，未登录再跳登录

**理由**:
- 不做 metadata，公开访问的增长价值会被砍掉一半。
- 继续分享站内链接，会让社区内容只能做“内链”，无法承担传播入口。

### D4: Remix 基于 publication snapshot 创建新项目，而不是回写原 conversation

**选择**: 新增 Remix 服务端流程：从 `publication_snapshots` 读取消息快照、ops 快照和画布状态，创建新的 `project/document/conversation`，将快照写入用户自己的项目上下文，然后跳转到编辑器继续创作。

**备选方案**:
- A) 直接以只读模式打开原 publication 并允许用户在其上继续写
- B) 只复制封面图或最终画布，不复制对话过程

**理由**:
- A 会把公开内容和私有创作边界混淆，也不利于后续权限隔离。
- B 虽然简单，但失去 Fluxa 的差异化资产：对话过程和设计过程。
- 基于 snapshot 创建新项目，与当前“项目是创作边界”的模型一致，也最符合“Fork/Remix”的用户心智。

### D5: 内容治理首版采用“举报 + 限频 + 运营隐藏”的最小闭环

**选择**:
- 新增举报对象：作品、评论、创作者资料
- 新增基础限频：评论频率、重复举报、匿名公开访问下的视图/交互防刷
- 新增最小处理能力：运营可将作品/评论状态改为 hidden / removed，举报记录进入待处理队列

**备选方案**:
- A) 首版不上治理，只依赖手工 SQL 处理
- B) 直接建设完整审核后台、自动审查、敏感词和封禁系统

**理由**:
- A 对正式发布不够安全，尤其你已经有评论和公开作者页。
- B 对当前阶段过重。
- “最小治理闭环”足够覆盖首发风险，同时为后续升级保留清晰接口。

### D6: 漏斗埋点采用统一事件模型 + 可替换 sink，不把 console 当正式方案

**选择**: 定义统一的发布漏斗事件，例如 `public_page_view`、`public_publication_view`、`remix_clicked`、`signup_started`、`signup_completed`、`first_project_created`、`first_generation_completed`、`first_publication_created`、`checkout_started`、`checkout_succeeded`。事件采集层保持一层封装，开发环境允许 console，生产环境必须可切到真实分析服务。

**备选方案**:
- A) 继续沿用零散 console log / metrics log
- B) 每个页面自己接各自的第三方 SDK

**理由**:
- A 无法支撑正式发布后的产品判断。
- B 容易造成事件命名混乱和重复上报。
- 统一事件模型最利于后续分析、A/B 和归因。

### D7: 公开首页升级为“内容驱动转化首页”，而不是再做一个纯营销页

**选择**: 在现有 [`src/app/page.tsx`](/Users/cj/Documents/personal/project/fluxa/src/app/page.tsx) 基础上，加入三类内容：价值主张、公开作品示例、明确 CTA。首页不需要承担完整 discover 能力，但必须承接“看示例 -> 注册 -> 开始创作”这条最短路径。

**备选方案**:
- A) 继续保持当前极简视觉页
- B) 直接把 discover 作为首页

**理由**:
- A 转化解释力不足。
- B 会削弱品牌表达，也让首页失去明确叙事。
- 公开首页更适合作为“转化页 + 精选入口”，discover 作为深入浏览页。

## Risks / Trade-offs

**[公开路由与站内路由并存会增加维护成本]** → **Mitigation**: 公共展示组件尽量复用，公开页和站内页只在 layout、权限增强态和 CTA 上分层。

**[Remix 复制快照会增加存储和数据复制量]** → **Mitigation**: 首版接受项目级复制成本；后续若数据量显著增长，再为快照导入增加压缩或延迟复制策略。

**[公开页面暴露后，社区垃圾内容和评论滥用风险上升]** → **Mitigation**: 上线前必须一并交付举报、限频和隐藏机制，且支持快速手工处理。

**[SEO 改造可能与当前客户端页面结构冲突]** → **Mitigation**: 公开入口尽量采用服务器组件负责首屏和 metadata，交互增强放到客户端子组件。

**[分析服务接入会带来额外依赖和隐私合规要求]** → **Mitigation**: 事件层先抽象，再按部署环境选择 sink；隐私政策同步补充分析与 cookie 说明。

**[首页改版可能影响现有品牌视觉简洁度]** → **Mitigation**: 保留当前视觉语言，但增加有限且高信号的转化模块，不做冗长营销堆砌。

## Migration Plan

1. 新增公开路由壳和公开页面入口，但先通过 feature flag 控制对外暴露。
2. 抽离发现页、作品详情页、创作者主页中可复用的公开展示组件，保持站内版本不受影响。
3. 为公开内容页补齐 metadata、canonical、分享 URL，并将编辑器分享入口切到真实公开链接。
4. 实现 Remix 创建链路，先覆盖作品详情页 CTA，再逐步扩展到更多入口。
5. 增加举报表/API、限频策略和最小运营处理流，并在正式放量前完成人工演练。
6. 接入统一漏斗事件与生产 sink，先验证核心事件链路，再用于首页与公开页转化观察。
7. 灰度开放公开内容页，再开放首页改版与 Remix，最后根据风控情况逐步放量。
8. 回滚策略：关闭公开入口与 Remix feature flag，保留已有发布数据；公开页面可降级到 404/登录跳转，治理与事件表保留不回退。

## Open Questions

1. 公开作品 URL 是否沿用 UUID，还是引入更可读的 slug；首版建议先沿用 UUID，避免发布链路再引入唯一 slug 约束。
2. Remix 首版是否要求登录后才能执行；建议是，未登录用户点击后先进入注册/登录，再回跳执行。
3. 举报处理入口是放在站内 profile/settings 还是先用轻量运营页面；建议先做轻量运营页或受控 SQL/RPC 流，不为首版搭完整后台。
4. 分析服务最终选型是否已有既定平台；若未定，先保留可替换 sink 抽象，不把具体供应商写死到 spec。

## Context

当前系统已具备用户认证与会员有效期字段（`user_profiles.membership_expires_at` 等），并已有支付路径授予会员权益。该变更需要在不侵入支付主流程的前提下，新增“邀请码赠送固定 Pro 30 天”的能力，并支持两条入口：注册时选填（延迟兑换）与个人页手动兑换（兜底）。

约束条件：
- 邀请码需要具备可审计与防滥用能力（一码一次、单用户限领一次）。
- 注册填写邀请码不应因邮箱未验证或首次登录时序问题造成提前消耗。
- 自动兑换失败不能阻塞用户进入 `/app`。

## Goals / Non-Goals

**Goals:**
- 支持邀请码兑换固定权益（Pro 30 天）。
- 注册流程支持邀请码选填，并在首次进入 `/app` 时自动尝试兑换。
- 提供个人页手动兑换入口。
- 在数据库层保证并发安全与可追溯审计。
- 统一兑换错误语义，便于前端稳定展示。

**Non-Goals:**
- 不实现邀请码准入门槛（无邀请码仍可注册与使用）。
- 不实现多档权益模板（7/30/90 天等）与活动后台运营面板。
- 不改造现有支付订单与结算流程。
- 不在本次引入复杂风控（设备指纹、IP 规则、频控策略中心）。

## Decisions

1) 兑换模型采用“单用途邀请码 + 审计流水”
- 决策：新增 `invite_codes`（状态机）与 `invite_code_redemptions`（审计）两张表。
- 理由：将“可用性判断”和“历史追踪”解耦，便于约束与运营分析。
- 备选方案：仅在一张表记录使用状态。缺点是审计字段膨胀且后续扩展（多次尝试、多种奖励）困难。

2) 注册时仅保存待兑换值，不立即核销
- 决策：在注册请求中接收邀请码（选填），写入用户 metadata 的 `pending_invite_code`，不直接兑换。
- 理由：避免邮箱验证前、未进入产品前就消耗邀请码，降低误用与客服成本。
- 备选方案：注册即兑换。缺点是时序脆弱，易出现“注册成功但未激活账号已消耗码”。

3) 兑换由受控服务端接口统一进入，核心逻辑在 DB RPC 内原子执行
- 决策：前端调用 API route，API route 再调用 `redeem_invite_code` RPC；RPC 负责校验、核销、授权、审计、会员更新。
- 理由：保证并发安全（事务 + 行级锁）、避免客户端直连敏感表、统一返回错误码。
- 备选方案：在应用层串行执行多条 SQL。缺点是并发竞争窗口更大，逻辑分散。

4) 会员授予策略采用“向后延长”
- 决策：`membership_expires_at = greatest(current_expires_at, now()) + interval '30 days'`。
- 理由：已是会员用户兑换时不丢剩余时长；过期用户从当前时间起算。
- 备选方案：固定覆盖为 `now()+30d`。缺点是可能吞掉现有权益，引发争议。

5) 错误语义标准化
- 决策：返回固定错误码：`INVALID_CODE`、`CODE_USED`、`CODE_EXPIRED`、`ALREADY_REDEEMED`、`NOT_AUTHENTICATED`、`INTERNAL_ERROR`。
- 理由：前端展示和埋点稳定，避免依赖底层错误字符串。
- 备选方案：直接透传数据库错误。缺点是耦合高且不可控。

## Risks / Trade-offs

- [风险] 用户在多端同时触发自动兑换导致重复提交 → **缓解**：RPC 内使用事务与锁，邀请码状态与“单用户限领”唯一约束双保险。
- [风险] 用户 metadata 残留导致重复尝试 → **缓解**：自动兑换成功后立即清理 `pending_invite_code`；失败时按错误类型决定是否清理（无效/已用/过期清理，临时错误保留）。
- [风险] 错误提示不一致影响体验 → **缓解**：只使用标准错误码映射文案，禁止前端直接展示底层异常。
- [取舍] 本次不做多权益模板，扩展性有限 → **缓解**：审计表保留 `benefit_type` 字段，为后续活动化扩展预留路径。
- [取舍] 自动兑换放在 `/app` 入口，首次进入多一次后端调用 → **缓解**：仅在存在 `pending_invite_code` 时触发。

## Migration Plan

1. 新增 migration：创建 `invite_codes`、`invite_code_redemptions`、索引与 RLS 策略。
2. 新增 migration：创建 `redeem_invite_code(input_code text)` RPC（安全定义与权限控制）。
3. 新增服务端 API route：统一调用 RPC，返回标准错误码。
4. 修改注册页与 AuthDialog：新增邀请码选填字段并写入 `pending_invite_code`。
5. 修改 `/app` layout：登录后检测并自动兑换 pending code（非阻塞）。
6. 修改个人页：新增手动兑换入口。
7. 灰度发布：先在内测环境导入少量邀请码验证；确认后全量。

回滚策略：
- 应用回滚：关闭自动兑换调用与前端入口（功能开关或回滚 commit）。
- 数据回滚：保留已产生审计记录，仅停用兑换接口与将邀请码批量置为 disabled；不删除历史流水。

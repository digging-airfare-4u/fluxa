# 灵感发现详情改为弹框设计

## 背景

用户明确要求：在“灵感发现”区域中，点击卡片时不再跳转到独立详情页，而是在当前页面内打开详情弹框。实现完成后，独立详情页应删除，只保留弹框交互。

当前实现中：

- 卡片组件 `src/components/discover/PublicationCard.tsx` 通过链接跳转到 `/app/discover/[id]`
- 详情内容由 `src/app/app/discover/[id]/page.tsx` 承载
- 首页灵感区 `src/app/app/page.tsx` 与 Discover 页 `src/app/app/discover/page.tsx` 都复用了 `PublicationCard`

## 目标

本次改动的目标很明确：

1. 点击灵感卡片时，在当前页面打开详情弹框
2. 弹框内保留现有详情页的大部分信息与能力
3. 不再依赖独立详情路由承载主要详情浏览
4. 删除不再需要的详情页文件，避免页面与弹框双轨并存

## 方案对比

### 方案 A：完整详情弹框（已选定）

将当前 `/app/discover/[id]` 页面中的详情内容提取为可复用详情视图，在卡片点击后使用 `Dialog` 弹出。

**优点**
- 最符合用户“不要跳页，只要弹框”的要求
- 可以最大程度复用现有详情展示逻辑
- 首页和 Discover 页都能获得一致体验

**缺点**
- 弹框承载内容较多，需要控制布局与滚动体验

### 方案 B：轻量预览弹框

只展示封面、标题、作者和少量操作，将评论、相关推荐、对话回放等内容移除。

**优点**
- 结构更轻，改动较少

**缺点**
- 无法保留原详情页核心能力
- 与用户希望保留详情体验的方向不一致

### 方案 C：弹框 + 完整页并存

卡片先打开弹框，但仍保留完整详情页入口。

**优点**
- 兼容两种浏览方式

**缺点**
- 会造成双套入口与维护成本
- 不符合“只要弹框”的明确要求

**结论：采用方案 A。**

## 最终设计

### 1. 交互方式

所有使用 `PublicationCard` 的灵感浏览场景，卡片点击后都不再执行 `router.push` 或 `<Link>` 跳转，而是改为打开详情弹框。

详情弹框应支持：
- 打开/关闭
- 根据当前选中的 publication id 加载详情数据
- 在当前页面上下文中完成浏览，不打断用户的滚动位置

### 2. 详情内容复用方式

当前 `src/app/app/discover/[id]/page.tsx` 中的核心详情内容将被拆分为可复用组件，供弹框使用。

弹框内应尽量保留以下内容：
- 封面图
- 标题、分类、描述
- 发布时间、浏览量、点赞、收藏等统计
- 点赞与收藏按钮
- 作者信息卡片
- 对话回放
- 评论区
- 相关推荐

这样可以保持原有详情能力，只改变承载容器。

### 3. 弹框布局

详情内容较长，因此弹框不应做成小尺寸居中卡片，而应采用更适合内容浏览的大尺寸弹框：
- 桌面端使用较宽的 `DialogContent`
- 高度限制在视口内
- 内容区内部滚动
- 顶部保留明确关闭能力

目标是让它像“页面级弹层”，而不是简单提示框。

### 4. 数据流

弹框打开后，根据传入的 publication id 拉取详情数据。

需要保留当前详情页中的数据行为：
- `fetchPublicationDetail`
- `fetchPublicationSnapshot`
- `fetchRelatedPublications`
- `fetchPublicProfile`
- `checkUserInteractions`
- `incrementViewCount`

数据加载逻辑应从页面组件中抽离，避免页面删除后逻辑丢失。

### 5. 相关推荐行为

相关推荐中如果继续复用 `PublicationCard`，需要避免再次跳页。

推荐做法是：
- 在弹框内部点击相关推荐卡片时，直接切换当前弹框的 publication id
- 不关闭弹框
- 不跳转页面

这样体验更连续，也符合“始终在弹框内浏览”的目标。

### 6. 页面删除策略

完成弹框化后，应删除不再需要的详情页文件：

- `src/app/app/discover/[id]/page.tsx`

如果项目中还有直接依赖该路由的跳转入口，也需要同步改为弹框方案，避免残留死路径或无用页面。

### 7. 兼容范围

本次优先覆盖以下已确认场景：
- 首页灵感区中的卡片点击
- Discover 页面中的卡片点击

如果 `PublicationCard` 在个人主页、作者主页或相关推荐区域中也被复用，则应统一使用同一弹框机制，保证交互一致。

## 风险与约束

### 风险 1：详情逻辑原本绑定在页面中
需要先抽离出可复用的详情内容和数据加载逻辑，否则页面删除后难以复用。

### 风险 2：弹框内部内容较长
评论区、对话回放、相关推荐都在同一弹框中，必须控制内部滚动和布局宽度，避免内容挤压。

### 风险 3：嵌套点击行为
相关推荐区域和外部卡片都可能再次触发详情切换，需要确保状态切换明确，不出现重复弹框。

## 范围边界

本次不做：
- Discover 数据查询接口重写
- 新的详情页视觉重设计
- 新增分享、下载等额外动作
- 将弹框改为抽屉、侧边栏或全屏路由

本次只聚焦：
- 卡片点击改为弹框
- 详情页内容迁移为弹框内容
- 删除独立详情页

## 设计结论

最终方案为：

**将灵感发现卡片的详情浏览从独立页面切换为当前页详情弹框，通过抽离现有 `/app/discover/[id]` 的详情逻辑为可复用组件来承载内容，并在完成迁移后删除独立详情页文件。**
